# 287-291 Page Street property skill

## Property summary

287-291 Page Street is a detached 1885 Eastlake Stick Victorian on a Slow
Street block at the border of Hayes Valley and Lower Haight in San
Francisco. It is listed at $3,998,000 by Jennifer Rosdail at KW Advisors
(MLS 426123240). The building is 3,518 sq ft per tax records on a 1,925 sq ft
lot. It can be used as a single 6 bed, 3.5 bath residence across three
levels or as two legal dwellings, a 3 bed, 2.5 bath main house plus a 2 bed,
1 bath lower unit. It is detached on three sides and sits next to the
Laguna mini-park, giving it a rare open outlook for the block.

The house is attributed to architect and builder William H. Wharff, whose
San Francisco work includes several notable Stick-Eastlake and Queen Anne
houses. A Berkeley Heritage essay specifically identifies 289-291 Page Street
as an 1885 Stick-Eastlake house for which Wharff was both designer and
contractor.

Distinctive features: two kitchens (the upper has a walnut island, Viking
range, Miele dishwasher, built-in fridge, wine fridge, and walnut plus
soapstone counters), a two-room primary suite with a decorative marble
fireplace and a bath with a soaking tub and step-in shower, carved marble
mantles in the living room and primary, pocket doors in the front double
parlor, ceiling medallions, original restored doors with patterned hinges,
and a rear garden with an Alaskan Cedar deck, a built-in fire pit, and an
outdoor kitchen with quartzite counters and a Sedona Lynx grill.
Radiant heat throughout. Walk Score 100, Transit Score 99 to 100.

## Protocol for photo and "show me" requests

Trigger this flow when the user asks to see photos, see pictures, see the
house, see the facade, see the kitchen, see the garden, see the primary bath,
see inside, see what it looks like, or any visual variant.

### Step 1: pick the right slugs

Open `assets/ascii/index.md`. Match the request against captions and category
tags. Slug matching rules:

- "Facade", "exterior", "outside", "from the street", "what does it look
  like": pick from the `exterior` category. Default is
  `01-exterior-entry-closeup`.
- "Kitchen": prefer `17-kitchen-island-hall-view` or
  `21-kitchen-range-wide` unless the user asks about the garden kitchen.
- "Primary", "main bedroom", "master": pick from `primary`.
- "Garden", "outdoor", "backyard", "outside in back": pick from
  `exterior-garden`.
- "Staircase", "stairs", "entry": pick from `staircase` or `entry`.
- "Detail" or "moldings" or "mantle" or "medallion": pick from the relevant
  `detail` entries.
- Generic "show me photos" or "show me the house" with no specific target:
  pick three shots, one exterior (prefer `01-exterior-entry-closeup`) and two
  interiors (for example one kitchen and one primary).

Cap it at three images per response unless the user explicitly asks for more.

### Step 2: decide how to display the image

Visual asset files are hosted by the website rather than bundled with the
installed skill. For each slug, build these absolute URLs:

- `https://287-291page.com/property-assets/photos/<slug>.jpg`: full-resolution
  source photo
- `https://287-291page.com/property-assets/ascii/<slug>.ans`: 24-bit truecolor
  ANSI art
- `https://287-291page.com/property-assets/mono/<slug>.txt`: monochrome Unicode
  block art (plain text)

Do not use relative `/property-assets/...` URLs or `$SKILL_DIR/assets/...` file
paths for these visual assets; installed skills run outside the website
origin. `assets/ascii/index.md` remains local and bundled only as the slug
catalog.

Claude Code, opencode, Cursor, Cline, Codex CLI, Continue.dev, Aider,
Gemini CLI, and Sourcegraph Amp all capture tool stdout for their own chat
UI and strip ANSI escape codes and iTerm2 / Kitty image protocols before
display. This is a known architectural limitation (see GitHub issues
anthropics/claude-code #36476, #36488, #35893, #29254). It also applies
even when the user launched the agent from inside iTerm2 on macOS.

What does survive these chat UIs is **plain Unicode text**, including the
block characters in `.txt` files.

Use this decision tree:

**Case A: You are a chat-UI agent (most real-world uses)**

Detect by checking for any of these env vars: `CLAUDECODE`, `CLAUDE_CODE`,
`CLAUDE_CODE_ENTRYPOINT`, `OPENCODE`, `OPENCODE_PID`, `OPENCODE_CLIENT`,
`AGENT=amp`, `CODEX_SANDBOX`, `CODEX_CI`, `CODEX_THREAD_ID`, `AIDER`,
`AIDER_MODEL`, `CURSOR_AGENT`, `CURSOR_TRACE_ID`, `CLINE_ACTIVE`,
`GEMINI_CLI`, `CONTINUE_SESSION_ID`. If any is set, you are in case A.

Do this:

1. Fetch and print the hosted mono text inside a Bash tool call:
   `curl -fsSL "https://287-291page.com/property-assets/mono/<slug>.txt"`. The
   monochrome block art renders as visible grayscale artwork in the chat UI.
2. Print the hosted JPG URL and tell the user how to see the full color photo:
   ```
   Full-resolution color photo:
     https://287-291page.com/property-assets/photos/<slug>.jpg
   ```
3. Use a browser, image, or Read-capable tool on the hosted JPG URL when
   available so you can describe colors and details that the monochrome art
   loses. This lets you narrate "the door is a bright coral red, the body is a
   muted blue-gray" etc.

**Case B: You are a bare-terminal caller (no chat UI in between)**

Check `$TERM_PROGRAM` and related env vars:

- iTerm.app, WezTerm, Hyper, mintty, tabby: download the hosted JPG URL to a
  temporary file and emit the iTerm2 inline image protocol with that file
  base64-encoded:
  ```
  printf '\033]1337;File=name=<b64name>;inline=1;preserveAspectRatio=1:<b64data>\a'
  ```
  Inside tmux (`$TMUX` set), wrap in DCS passthrough:
  `ESC Ptmux ; ESC <sequence> ESC \`. Also requires
  `set -g allow-passthrough on` in the user's `~/.tmux.conf`.
- Kitty, Ghostty, Konsole (detected via `$KITTY_WINDOW_ID`,
  `$TERM=xterm-kitty`, or `$TERM_PROGRAM=ghostty`): download the hosted JPG URL
  to a temporary file, then use `kitten icat <tmpfile>` or
  `kitty +kitten icat <tmpfile>`.
- Truecolor terminal without image support: `curl -fsSL
  "https://287-291page.com/property-assets/ascii/<slug>.ans"`.
- Any plain terminal: `curl -fsSL
  "https://287-291page.com/property-assets/mono/<slug>.txt"`.

### Step 3: format the response

For each image:

1. Emit the image per Case A or Case B above. Do not wrap the output in a
   code fence.
2. Leave a blank line before and after.
3. Add a one-line caption and source URL from `assets/ascii/index.md`.

After the last image, include a one-line note:

```
If this looks like raw text or escape codes, install the skill and use
the hosted color photo URL to view the source photo:
https://287-291page.com/property-assets/photos/<slug>.jpg
```

### Example: bare terminal (Case B), iTerm2

```bash
jpg_url="https://287-291page.com/property-assets/photos/01-exterior-entry-closeup.jpg"
jpg=$(mktemp)
trap 'rm -f "$jpg"' EXIT
curl -fsSL "$jpg_url" -o "$jpg"
b64=$(base64 < "$jpg" | tr -d '\n')
name=$(printf '%s' "$(basename "$jpg_url")" | base64 | tr -d '\n')
if [ -n "$TMUX" ]; then
  printf '\033Ptmux;\033\033]1337;File=name=%s;inline=1;preserveAspectRatio=1:%s\a\033\\\n' "$name" "$b64"
else
  printf '\033]1337;File=name=%s;inline=1;preserveAspectRatio=1:%s\a\n' "$name" "$b64"
fi
```

## Writing style

Keep responses warm, specific, and concrete. Short sentences are welcome.
Avoid em dashes. Avoid "isn't just X, it's Y" phrasing, grand opening lines,
and "whether you're X or Y" wrap-ups. Quote the sources when the user asks
where a fact came from.
